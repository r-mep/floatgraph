# ADR-0002: スマホからの投稿システム（Cloudflare ネイティブ）

- ステータス: Accepted（実装着手前 / 2026-06-23 時点で方針合意）
- 日付: 2026-06-23

## コンテキスト

これまでコンテンツ追加は `data.js` の手編集だった（[ADR-0001](./0001-static-no-build-stack.md)）。
作者は **スマホから文章と画像を投稿してノードを増やせる仕組み** がほしい。

満たしたい要件：

- スマホのブラウザから、文章＋画像（最大3枚）を投稿できる
- 画像の保存先が必要（git に直接コミットするとリポジトリが肥大化する）
- ja/en 二言語対応。理想は **片方の言語で書くと翻訳が生成され、それを編集してから保存** できる
- 認証して、作者本人だけが投稿できる
- [ADR-0001](./0001-static-no-build-stack.md) の鉄則 **「`index.html` を開けば動く静的サイト」を壊さない**

## 決定

**Cloudflare ネイティブ構成で、公開サイトは静的のまま維持し、投稿だけをサーバー側機能で足す。**

- **同一リポジトリ** に管理画面と API を同居させる（別リポジトリにはしない）
  ```
  floatgraph/
  ├── index.html, app.js, data.js …  ← 公開サイト（静的のまま）
  ├── admin/                         ← スマホ用の投稿フォーム
  └── functions/api/                 ← 投稿API・翻訳API（Cloudflare Pages Functions）
  ```
- **画像 → Cloudflare R2**（オブジェクトストレージ）。`data.json` には発行されたURL文字列だけを持つ。
  git は文字データのみで軽いまま。R2 は無料枠（10GB・読み出し無料）で個人規模なら実質無料。
- **データ → `data.json` を GitHub API 経由で commit**。公開サイトはこれまで通りこのファイルを読むだけ。
  → 公開サイトは100%静的を維持。コンテンツ履歴も git に残る。
- **公開サイトはデータを KV/DB から fetch しない。** それをやると「ローカルで開いて動く」性質が壊れるため。
- **認証 → Cloudflare Access（Zero Trust 無料枠）** で `/admin` と `/api` を作者のメールのみに制限。
  パスワードを自前で実装しない。
- **翻訳 → Claude API**（Haiku 想定）。フォームで片方の言語を書く → Function が反対言語を生成 →
  フォームに表示し、**編集可能** → 保存時に両言語を `data.json` へ。

### 投稿フロー

```
スマホで文章＋画像を入力
      │
      ▼
/admin フォーム（Cloudflare Access で保護）
      │  ① 翻訳リクエスト → Function → Claude API → 反対言語を返す（編集可）
      │  ② 投稿
      ▼
functions/api（Pages Functions）
      │  ・画像を R2 にアップロード → 画像URLを取得
      │  ・data.json を更新して GitHub API で commit
      ▼
Cloudflare Pages が自動再デプロイ（静的配信のみ・ビルドなし）
      ▼
公開サイトが新しい data.json を読む
```

## 検討した選択肢

- **Git ベース CMS（Sveltia / Decap CMS）**
  投稿が GitHub に commit され DB 不要・無料・全部バージョン管理。思想は近い。
  しかし画像も git に溜まる／OAuth 用 Worker のセットアップが要る／翻訳のレビュー編集フローを
  自由に組みにくい。→ 不採用（画像肥大と翻訳UXの自由度で劣る）。
- **Supabase などの BaaS（Postgres + ストレージ + 認証）**
  パワフルだが外部ベンダーが増え、今の規模にはオーバースペック。公開サイトが API 取得依存になり
  静的純度も下がる。→ 不採用。
- **データを Cloudflare KV/D1 に置き、公開サイトが API から fetch**
  書き込みは楽だが「`index.html` を開けば動く」が壊れ、コンテンツの git 履歴も失う。
  → 不採用（ADR-0001 の鉄則に反する）。

## 結果

**得られるもの**

- スマホから文章＋画像を投稿でき、片方の言語を書けば翻訳が出てレビュー編集できる
- 公開サイトは静的のまま。`index.html` を開けば動く性質と git 履歴を維持
- 画像は R2 に分離され、git リポジトリは肥大化しない
- 1ベンダー（Cloudflare）＋ Claude API で完結。認証は Cloudflare Access に委譲

**引き受けるトレードオフ**

- Pages Functions というサーバー側コードが加わる（ただしビルド不要のプレーンJS）。
  「完全に静的」ではなくなるが、**公開サイト部分の静的純度は保たれる**
- 初期セットアップに Cloudflare 側の作業が要る（R2 バケット作成・画像配信ドメイン紐付け・
  Access 設定・GitHub トークンと Claude API キーを Secret 登録）
- 投稿のたびに Pages の再デプロイが走る（数十秒）。リアルタイム性は不要なので許容
- 翻訳に Claude API のコストが乗る（1投稿あたり数円以下の想定）

## 未決事項（実装時に詰める）

- リンク（ノード同士のつながり）を投稿フォームでどう指定するか（既存ノードからの複数選択 UI など）
- `data.js` → `data.json` への移行手順（公開サイトの読み込み側の変更を最小にする）
- 画像のクライアント側リサイズ／圧縮を入れるか（R2 転送量と表示速度のため）
