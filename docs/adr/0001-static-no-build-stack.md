# ADR-0001: ビルドツールなしの静的サイト構成

- ステータス: Accepted
- 日付: 2026-06-23（プロジェクト初期からの方針を遡って記録）

## コンテキスト

floatgraph は個人サイト（Obsidian グラフビュー風の浮遊ノード）。
作者ひとりで長く緩く保守していくことが前提で、次のことを重視したい。

- **寿命の長さ** — 数年後に開いても壊れていないこと。npm 依存やビルド設定は時間が経つと腐りやすい
- **手数の少なさ** — 「ファイルを開けば動く」「push すれば公開される」だけで完結させたい
- **学習・改変のしやすさ** — フレームワークの作法を覚えなくても、HTML/CSS/JS を直接いじれば変えられる

サイトの規模は小さく（ノード数十、ページ実質1枚）、動的なサーバー処理も当初は不要だった。

## 決定

**ビルドツールを一切使わない、純粋な静的サイトとして構築する。**

- 言語：HTML / CSS / JS のみ。`index.html` をブラウザで直接開けば動く
- 依存ライブラリ：CDN 経由のみ（D3.js）。`node_modules` も `package.json` も持たない
- データ：`data.js` にプレーンな JS オブジェクトとして保持。ここだけ編集すれば内容が変わる
- 国際化：UI 文字列は `i18n.js`、コンテンツは `data.js` 内で `{ ja, en }` オブジェクトとして持つ
- テーマ：色は CSS 変数（`var(--bg)` など）に集約し、`:root` / `[data-theme]` で切り替え
- ホスティング：Cloudflare Pages。`main` への push で自動デプロイ（ビルドステップなし）
- ドメイン：`floatin.mep-std.com` → CNAME → `floatgraph.pages.dev`

ファイルの役割分担：

| ファイル | 役割 |
|---|---|
| `index.html` | DOM 構造 |
| `style.css` | 全スタイル・テーマ変数 |
| `i18n.js` | UI 固定文字列（ja/en） |
| `data.js` | ノードとリンクのデータ |
| `app.js` | D3・パネル・言語・テーマのロジック（IIFE で囲んでグローバル汚染を防ぐ） |

## 検討した選択肢

- **Vite + フレームワーク（React/Vue/Svelte）**
  コンポーネント化や状態管理は楽になるが、ビルド設定・依存管理・ツールチェーンの保守コストが
  サイトの規模に見合わない。数年後にビルドが通らなくなるリスクも高い。→ 不採用。
- **静的サイトジェネレータ（Astro/11ty 等）**
  Markdown 管理などは魅力だが、同じくビルド前提になる。ページが実質1枚で恩恵が薄い。→ 不採用。
- **素の HTML/CSS/JS（採用）**
  機能は自前で書く必要があるが、規模が小さいため十分に管理可能。寿命・手数・改変性すべてで有利。

## 結果

**得られるもの**

- `index.html` を開くだけで動く。ローカル開発はサーバーすら不要（`python3 -m http.server` で十分）
- ツールチェーンの腐敗リスクがほぼゼロ。依存は CDN の D3 のみ
- デプロイは `git push` だけ。ビルド時間なし
- コードが透明で、HTML/CSS/JS が読めれば誰でも改変できる

**引き受けるトレードオフ**

- コンポーネント再利用や型安全の恩恵はない。共通化は関数とコメント規約で担保する
- コンテンツ追加は `data.js` の手編集。スマホからの投稿などはこの構成だけでは不可
  → この制約を解消するため [ADR-0002](./0002-mobile-posting-system.md) で投稿システムを追加する
- データが静的なため、動的な機能（全文検索のサーバー処理など）は別途仕組みが要る

**この方針は意図的なものであり、原則として変えない。** 機能追加は「静的サイトの純度を保ったまま
拡張できるか」を first principle として検討する。
